# 食光记 · AI PM 作品集架构

> 每个标题下标注了 **PM 产出物**、**产出形式**、**受众**，让你拿到架构就能动笔。

---

## 一、产品定义

### 1.1 需求背景与核心痛点

**产出物**：一段 300-500 字的背景叙述 + 痛点清单（3-5 条）。

**产出形式**：文字 + 要点列表。

**受众**：面试官 / 评审委员会。

**写什么**：
- 当前饮食记录市场用户的核心困扰是什么？（查食物库太累、坚持不下去、只看热量不看结构）
- 这些困扰为什么传统方案解决不了？
- 每一条痛点要有具体的场景支撑（"用户点了一碗牛肉面，MyFitnessPal 里找不到，就放弃了"），不要写抽象结论。
- 最后用一句话收束：为什么这个时机适合做这个产品。

---

### 1.2 产品定位与价值主张

**产出物**：一句话产品 slogan + 产品定位陈述（100 字以内）。

**产出形式**：文字。

**受众**：全团队 / 面试官。

**写什么**：
- 产品叫什么、为谁、解决什么问题、用什么方式。
- 核心差异化用一句话说清楚（例："用自然语言记录饮食，AI 帮你看到热量之外的东西"）。
- 与竞品的定位差异（一句话对比即可，详细竞品分析放 1.4）。

---

### 1.3 目标用户画像

**产出物**：2-3 个用户画像卡片，每个含：姓名、年龄、职业、饮食场景、核心诉求、当前痛点、使用动机。

**产出形式**：卡片式文字（每个画像半页）。

**受众**：全团队（设计、开发、运营都需要知道"为谁做"）。

**写什么**：
- 画像要有区分度（比如减脂白领 vs 增肌健身者 vs 控糖中老年），而不是 3 个差不多的"关注健康的年轻人"。
- 每个画像写清楚：一天中什么时候用这个产品、用的时候在什么环境、最想要的是什么。

---

### 1.4 竞品分析

**产出物**：竞品矩阵表 + 差异化定位图 + 关键洞察（2-3 条）。

**产出形式**：表格 + 2×2 定位图（横轴纵轴自己定，如"手动输入 ↔ AI 智能" × "热量中心 ↔ 营养结构中心"）+ 文字总结。

**受众**：全团队 / 投资人。

**写什么**：
- 竞品矩阵至少覆盖 5-6 个产品，分三类：传统卡路里计算型（MyFitnessPal、薄荷健康、Lose It!）、新兴 AI 型（如果有）、硬件+App 型。
- 每个竞品写清楚：核心功能、优势、劣势、目标用户、商业模式。
- 定位图的作用是视觉化展示"市场空白在哪里、我们在哪里"。
- 关键洞察回答：市场还有没有机会？最大的对手是谁？我们凭什么赢？

---

### 1.5 AI 可行性论证

**产出物**：一份判定表 + 文字论述（500 字以内）。

**产出形式**：表格 + 论述段落。

**受众**：技术负责人 / 评审委员会。

**写什么**：回答三个问题——
1. **为什么必须用大模型？** 列出所有你能想到的非 AI 替代方案（规则引擎、查数据库、模板匹配），逐一说明为什么不够。
2. **大模型不可替代的价值是什么？** 自然语言理解、模糊推理、个性化反馈生成——具体到场景。
3. **大模型的风险是什么？** 幻觉、成本、延迟——每个风险都要有对应的缓解策略。

判定表示例格式：

| 能力需求 | 规则引擎能解决？ | 传统 ML 能解决？ | 大模型能解决？ | 结论 |
|---------|:---:|:---:|:---:|------|
| 从"一碗牛肉面"估算营养素 | ✗ | ✗ | ✓ | 必须 LLM |
| 营养素目标范围计算 | ✓ | — | — | 规则即可 |
| 生成个性化饮食反馈 | ✗ | ✗ | ✓ | 必须 LLM |

---

### 1.6 预期目标

**产出物**：业务目标表 + AI 质量目标表。

**产出形式**：两张表格，每张 5-8 项，含指标名 + 目标值 + 衡量方式 + 时间节点。

**受众**：全团队 / 管理层。

**写什么**：
- **业务目标**：日活、留存（次日/7日/30日）、日均记录次数、付费转化率等——写清楚 MVP 阶段目标 vs 正式版目标。
- **AI 质量目标**：分析触发成功率、追问率、用户修正率、格式合规率、单次分析 token 消耗上限、p50/p95 响应时间。每个指标写清楚"为什么这个指标重要"。

---

## 二、用户场景与业务流程

### 2.1 核心用户场景设计

**产出物**：5-6 个用户场景故事，每个 200-300 字 + 场景亮点标注。

**产出形式**：叙事文字 + 亮点 callout。

**受众**：全团队（这是团队理解"产品要做什么"的最直观材料）。

**写什么**：每个场景写成一个小故事——
- 用户是谁（关联 1.3 的画像）、在什么环境下、想做什么、怎么做的、得到了什么结果。
- 每个场景末尾标注 **场景亮点**（这个场景里 AI 的价值体现在哪里，与传统产品有何不同）。
- 6 个推荐场景：
  1. 日常快速记录（"吃完饭随手说一句"）
  2. 外食场景（拍食物照片识别）
  3. 周度健康回顾（"这周我吃得怎么样"）
  4. 目标驱动场景（减脂/增肌/控糖模式下 AI 反馈策略不同）
  5. AI 营养助手被动咨询（用户主动提问，AI 结合个人数据回答）
  6. AI 营养助手主动关怀（AI 检测漏餐主动询问、建议监督提醒）

---

### 2.2 用户旅程与交互流程

**产出物**：3 张用户旅程图 + 3 张交互流程图 + 关键页面线框图。

**产出形式**：图表（Figma / ProcessOn / Excalidraw 等工具绘制）+ 配套说明文字。

**受众**：前端开发 + UI 设计师 + 测试。

**写什么**：
- **用户旅程图**（User Journey Map）——不是页面跳转，是一条时间线。每个旅程包含阶段、用户行为、情绪曲线（高/低点标注）、痛点、机会点。3 条旅程：
  - 新用户引导（发现产品 → 注册 → 填写资料 → 首次记录 → 看到第一次分析）
  - 日常记录（打开 App → 输入 → AI 判断/追问 → 看到结果 → 确认保存）
  - 回顾分析（打开首页 → 看每日总结 → 进入历史 → 看周报告）

- **交互流程图**（Interaction Flow）——页面级跳转逻辑，含所有分支。不同于旅程图，这是给开发看的，要把每个判断节点画清楚。例：
  ```
  [首页] → 点+号 → [聊天界面] → 输入文字/拍照
    → AI 信息检查 → 够？→ 分析结果卡片 → 确认 → 回首页
                  → 不够？→ 追问气泡 → 用户补充 → 回到信息检查
  ```
  每张图配一段文字说明关键设计决策（如"为什么信息检查和营养分析分两步而不是一步"）。

- **关键页面线框图**——不需要高保真，手绘或低保证即可。目的是让读者看到产品长什么样。

---

### 2.3 异常流程处理

**产出物**：异常处理策略表。

**产出形式**：表格。

**受众**：前端 + 后端 + 测试。

**写什么**：覆盖所有"出错"场景：

| 异常场景 | 触发条件 | 用户看到什么 | 用户能做什么 | 系统在做什么 |
|---------|---------|-------------|-------------|-------------|
| AI 超时 | 分析 >15s 无返回 | "正在分析中..." | 等待/取消重试 | 等待 or 超时断开 |
| 网络断开 | fetch 失败 | "网络连接异常" | 重试 | — |
| AI 格式异常 | JSON 解析失败 | "分析出错了" | 重试/手动记录 | 降级到 content 解析 |

每行说明清楚：文案由 PM 定（不是开发自己想），操作项由 PM 决定（重试？降级？放弃？）。

## 三、核心功能模块

### 3.1 功能全景脑图

**产出物**：一张功能脑图。

**产出形式**：思维导图（树形结构）。

**受众**：全团队。

**写什么**：
- 从产品核心出发，逐层展开到具体功能点。至少展开 3 层。
- 用标注区分：MVP 已有 / 近期规划 / 远期规划。
- 不要写成功能清单——脑图要体现功能之间的层级关系和归属。

---

### 3.2 功能模块拆解（按页面逻辑）

**产出物**：每个模块一份功能说明（300-500 字）+ 该模块的功能点列表。

**产出形式**：文字 + 列表。

**受众**：全团队（特别是设计和开发，这是他们拆任务的依据）。

**写什么**：按产品页面逻辑组织为 6 个模块，每个写清楚"这个模块对应哪个页面/入口、解决什么问题、核心功能点有哪些、AI 在里面的角色是什么"——

1. **首页（仪表盘）**：每日营养概览、今日饮食摘要卡片、各营养素达标状态一眼可见、快速记录入口（点 + 号直接进入记录流程）。AI 角色：基于当日已记录数据生成个性化问候语和简要提示（"早餐蛋白质偏少，午餐可以补一下"）。

2. **饮食记录**：文字描述记录（MVP 已有）、拍照识食（扩展）、语音输入（扩展）、扫码记录（扩展）。核心流程：用户输入 → AI 判断信息完整性 → 追问或直接分析 → 展示营养分析结果卡片 → 用户确认保存。AI 角色：理解自然语言食物描述、估算营养素、生成反馈。

3. **历史记录**：按日期分组查看过往饮食记录、支持搜索筛选、多选删除。MVP 已有基础查看和删除功能，扩展方向为搜索、筛选、编辑单条记录。AI 角色较轻（此页面以数据展示为主）。

4. **数据分析**：每周趋势分析（MVP 已有）、自定义周期分析（扩展）、营养趋势图表（MVP 有基础 SVG 图表）、AI 生成的周期洞察报告（扩展）。AI 角色：对比实际摄入 vs 目标范围、发现趋势和异常、生成自然语言洞察。

5. **AI 营养助手**：多轮健康咨询对话、个性化饮食知识科普、**主动智能提醒**——这是区别于传统聊天机器人的核心功能。AI 角色权重最大，分两块：
   - **被动响应**：用户主动提问（"我最近总是下午犯困和吃有关系吗？"），AI 结合用户目标和饮食记录给出个性化回答。
   - **主动触达**：AI 根据用户状态主动发起消息，而非等待用户来问。典型触发场景：
     - 漏餐关怀：检测到用户某餐未记录，在餐后 1-2 小时人性化询问"午饭吃了吗？吃了什么呀～"
     - 建议监督：午餐给过建议（如"蔬菜偏少"），在晚餐前后主动提醒"晚餐别忘了加点蔬菜哦"
     - 目标进度提醒：检测到连续几天某项营养素偏离目标，主动告知并给出调整建议
     - 里程碑鼓励：用户连续记录 7 天时主动肯定
   - 主动触达的产品边界：PM 需定义触发频率上限（如每天最多 2 条主动消息）、推送时段限制（如不打扰休息时间）、用户可关闭主动提醒。

6. **个人中心**：身体资料管理（身高/体重/年龄/目标）、目标模式切换（减脂/增肌/维持）、目标范围预览与手动调整、应用 AI 建议的下周目标、数据导出。AI 角色：每周分析后生成下周目标调整建议，用户可选择采纳或忽略。

---

### 3.3 关键页面原型

**产出物**：关键页面原型图（MVP 已有点截图，扩展功能画线框图）。

**产出形式**：截图 + 线框图 + 页面说明。

**受众**：设计师 / 面试官。

**写什么**：
- 列出全量页面清单（所有功能模块涉及的关键页面），标注哪些 MVP 已有截图、哪些需要画线框图。
- 每个扩展功能的页面至少给一张线框图。
- 重点是展示页面之间的层级关系和导航结构，不只是孤立的单页图。

## 四、大模型设计

> 本章是 AI PM 作品集的**差异化核心**——传统 PM 不做这些。分三个板块：AI 策略 → Agent 设计 → 质量保障。

### —— AI 策略 ——

### 4.1 模型选型与多模型策略

**产出物**：模型选型决策表 + 多模型路由策略说明。

**产出形式**：表格 + 文字（300 字以内）。

**受众**：技术负责人 / AI 工程师。

**写什么**：
- **主模型选择**：列出备选模型（DeepSeek、GPT-4o、Claude 等），从成本、延迟、中文能力、结构化输出可靠性、API 稳定性五个维度对比。给出选型结论和理由。
- **多模型路由策略**：简单任务（信息检查）用小模型，复杂任务（每周分析）用大模型，特殊任务（内容安全审核）用专用模型。画一张路由决策树。
- **模型版本管理**：模型升级时的测试策略、灰度切换方案。

---

### 4.2 AI 能力边界

**产出物**：AI 职责边界表。

**产出形式**：表格。

**受众**：全团队。

**写什么**：
- 列出所有能力点，分三列标注：**AI 负责** / **系统负责** / **边界模糊（需讨论）**。
- 每项标注决策理由。例：
  - 营养素估算 → AI（需要常识推理）
  - BMR/TDEE 计算 → 系统（有公式，不需要 AI）
  - 营养素目标范围 → 系统（公式计算）+ AI 微调建议（每周分析时）
- 边界模糊项重点说明取舍逻辑。

---

### 4.3 业务规则

**产出物**：业务规则清单。

**产出形式**：表格 + 文字说明。

**受众**：AI 工程师（写入 prompt）+ 法务/合规。

**写什么**：三类规则——
1. **营养分析约束**：AI 不诊断疾病、不替代医生建议、不推荐极端饮食方案、涉及"过敏""中毒"等关键词时的标准话术。
2. **内容安全规则**：识别和干预极端饮食行为（催吐、过度节食、暴食）的策略、AI 检测到风险时的标准响应文案。
3. **隐私与合规**：用户饮食数据的使用边界、AI 训练数据中不包含用户个人数据、数据删除策略。

---

### —— Agent 设计 ——

### 4.4 Agent 架构设计

**产出物**：Agent 架构决策说明 + Workflow vs Agent 选择矩阵 + Agent 决策流程图 + 工具清单表。

**产出形式**：表格 + 流程图 + 文字。

**受众**：AI 工程师 / 后端开发。

**写什么**：四个子内容——

**4.4.1 Workflow vs Agent 的选择矩阵**

| 场景 | 推荐模式 | 原因 |
|------|---------|------|
| 单餐营养分析 | Workflow | 流程确定，两步必走 |
| 信息完整性判断 | Workflow | 判断逻辑明确 |
| 每日/每周总结 | Workflow | 输入输出结构固定 |
| AI 营养助手对话 | Agent | 用户问题不可预测，需要多工具自主组合 |
| AI 营养助手主动触达 | 混合 | 触发条件由系统规则判断（定时检测漏餐/达标状态），触达后对话由 Agent 处理 |

**4.4.2 Agent 决策流程**

画一张图：用户输入 → Agent 推理 → 决定调哪个 tool → 获取结果 → 继续推理 or 输出回答。标记出 Agent 的自主决策范围 vs 系统强约束点。

**4.4.3 工具清单与定义**

完整的工具定义表：

| 工具名 | 用途 | 输入参数 | 输出内容 | 提供方 | 适用 Agent |
|--------|------|---------|---------|--------|-----------|
| `info_check` | 判断信息完整性 | query, history | isEnough, assistantMessage | LLM Tool | 饮食记录 |
| `meal_analysis` | 单餐营养分析 | foodText, userProfile | summary, nutrition_estimate, feedback | LLM Tool | 饮食记录 |
| `daily_summary` | 每日整体分析 | 当日三餐数据 + 目标范围 | daily_nutrition_status, feedback | LLM Tool | 每日总结 |
| `weekly_analysis` | 每周趋势分析 | 7天数据 + 周目标 | weekly_status, next_week_target | LLM Tool | 每周分析 |
| `get_today_meals` | 查询今日已记录餐食 | userId, date | 餐食列表 + 缺失餐次 | 后端 API | AI 营养助手 |
| `get_recent_meals` | 查询近 N 天饮食记录 | userId, days | 餐食列表 + 营养日均值 | 后端 API | AI 营养助手 |
| `get_user_profile` | 获取用户身体资料与目标 | userId | 身高/体重/年龄/目标模式/目标范围 | 数据库 | 多个 Agent |
| `search_food_nutrition` | 查询食物营养数据 | foodName | 每100g 营养成分 | RAG 知识库 | AI 营养助手 |
| `search_diet_knowledge` | 检索膳食指南知识 | query | 相关知识条目摘要 | RAG 知识库 | AI 营养助手 |
| `check_meal_missed` | 检测是否有漏餐 | userId, date, currentTime | 缺失餐次列表 | 后端 API | AI 助手主动触达 |
| `check_goal_status` | 检测营养素达标状态 | userId, date | 各项达标/偏离摘要 | 后端 API | AI 助手主动触达 |
| `send_proactive_message` | 发送主动触达消息 | userId, messageType, context | 消息内容 + 发送 | 后端服务 | AI 助手主动触达 |

---

### 4.5 System Prompt 设计

**产出物**：每个 Agent/Workflow 节点的 System Prompt 结构模板 + 版本管理说明。

**产出形式**：结构化文本（不是完整 prompt 全文，而是结构拆解）。

**受众**：AI 工程师 / Prompt Engineer。

**写什么**：每个 prompt 拆成四段写清楚——
1. **角色定义**：AI 是谁、用什么语气说话
2. **可用工具**：有什么工具、什么情况用哪个
3. **行为边界**：不能做什么、不能说什么
4. **输出格式**：指向哪个 Tool Schema、字段要求

各 Agent 的 Prompt 设计要点：
- **饮食记录 Agent**：约束追问行为（最多 2 个问题、≤20 字）、feedback 调性（不说教、不制造焦虑）、只分析当前这一餐
- **每日/每周总结 Agent**：不逐餐复述、状态判断规则（蛋白质充足/达标/不足 vs 碳水超标/均衡/不足）、下一周目标调整原则
- **AI 营养助手 Agent（被动咨询）**：角色为私人营养顾问、回答须结合用户个人数据和饮食记录、引用膳食指南知识库、不诊断疾病
- **AI 营养助手 Agent（主动触达）**：角色为温和的饮食伙伴、语气轻松有分寸、不催促不施压、每次触达只提一件事

外加一段 Prompt 版本管理的说明：每次改 prompt 记录什么（改动内容 + 改动原因 + 预期效果 + 实际效果），用什么方式管理（Git 管理代码内 prompt）。

### 4.6 Memory 设计

**产出物**：Memory 架构说明——短期记忆策略 + 长期记忆清单表 + 工作记忆检索策略。

**产出形式**：文字 + 表格。

**受众**：AI 工程师 / 后端开发。

**写什么**：三种记忆分开写——

**短期记忆**：
- 上下文窗口大小设定（多轮对话保留最近几轮）
- 记忆压缩策略（超出窗口时，摘要还是截断？）
- 不同场景的窗口策略（饮食记录对话 5 轮 vs AI 营养助手对话 15 轮）

**长期记忆清单表**：

| 记忆项 | 存储位置 | 更新时机 | 读取时机 | 优先级 |
|--------|---------|---------|---------|--------|
| 用户身体资料 | user_profile 表 | 用户手动修改 | 每次 AI 分析时 | 必读 |
| 用户目标范围 | targets 表 | 每周分析后更新 | 每日/每周分析时 | 必读 |
| 近 7 天饮食记录摘要 | meal_summary 表 | 每次记录后增量更新 | AI 助手对话、主动触达判断 | 必读 |
| AI 助手对话历史 | conversation_log 表 | 每次对话后追加 | AI 助手后续对话 | 必读 |
| 主动触达记录 | proactive_log 表 | 每次触达后写入 | 判断是否该再次触达（频控） | 必读 |
| 用户偏好（口味/忌口等） | user_prefs 表 | AI 推断+用户确认 | AI 助手回答时参考 | 选读 |

**工作记忆（检索策略）**：
- Agent 执行任务时临时拉取什么信息？
- 不同场景的检索范围（饮食记录只需要当天数据，周报告需要 7 天数据 + 上周目标，教练对话可能需要查知识库）
- 检索优先级：先查本地 → 再查 RAG → 最后让 LLM 估算

### 4.7 RAG 知识库构建

**产出物**：知识库架构说明——三个知识库的内容范围 + 检索策略表。

**产出形式**：文字 + 表格。

**受众**：AI 工程师 / 后端开发 / 数据团队。

**写什么**：
1. **食物营养数据库**：数据来源（中国食物成分表 / USDA 等）、覆盖范围、更新频率。这是基础数据层，精度最高。
2. **饮食指南知识库**：来源（《中国居民膳食指南 2022》等权威文献）、内容类型（饮食原则、营养素推荐量、特殊人群建议）。用于 AI 回答"应该怎么吃"类问题时有据可依。
3. **检索策略**：什么时候查哪个库？检索结果的排序和截断规则（返回 top-k 条，按相关性排序）。检索失败时的降级方案。

### —— 质量保障 ——

### 4.8 结构化输出设计

**产出物**：每个 Tool 的输出 Schema 字段清单 + JSON 示例 + 设计原则说明。

**产出形式**：字段定义表 + JSON 示例代码块 + 文字。

**受众**：AI 工程师 / 后端开发 / 前端开发。

**写什么**：
- **每个 Tool 的字段定义表**（如 `meal_analysis` 的输出字段：字段路径、类型、必填、枚举值、约束、含义说明）。
- **每个 Tool 的 JSON 示例**（一个正常的完整示例，一个边界情况的示例）。
- **设计原则说明**：
  - 枚举优先（能被限制的值不用自由文本）
  - 扁平优先（不深层嵌套，前端好解析）
  - Description 即 Prompt（Tool definition 里的 `description` 直接影响 LLM 输出质量）
  - 两套枚举语的语义差异（为什么蛋白质用"充足/达标/不足"，碳水用"超标/均衡/不足"）
- **机制选择**：什么场景用 `response_format`（简单 JSON），什么场景用 Tool Calling（复杂嵌套、可靠性要求高），什么情况加代码后处理，各自的优劣和适用条件。

### 4.9 幻觉防控与容错

**产出物**：幻觉防控策略表 + 容错规则清单。

**产出形式**：两张表格 + 文字。

**受众**：AI 工程师 / 后端开发。

**写什么**：
- **三层防控体系**：
  1. 输出前：Tool Calling 强制结构 + Schema 约束
  2. 输出后：代码校验（类型检查、枚举值检查、数值范围检查、字段完整性检查）
  3. 异常处理：重试 + 降级 + 兜底

- **校验规则表**：

| 校验项 | 规则 | 触发时处理 |
|--------|------|-----------|
| 格式合规 | JSON 可解析 | 解析失败→重试 1 次→降级到 content 提取 |
| 枚举值 | 值在允许集合内 | 不在→替换为默认值 |
| 数值范围 | 营养素在合理区间内 | 超标→标记异常但保留原值（可能是真的） |
| 字段完整性 | required 字段全部存在 | 缺失→给默认值并记录日志 |

- **容错降级链路**：Tool Calling 无返回 → 从 content 提取 JSON → JSON 不完整 → 重试一次 → 仍失败 → 展示错误信息 + 让用户手动记录

### 4.10 一致性检验

**产出物**：一致性要求定义表。

**产出形式**：表格 + 文字（200 字以内）。

**受众**：AI 工程师 / 测试。

**写什么**：
- 哪些场景下，相同的用户输入应该返回高度一致的结果？（营养分析、每日总结的一致性要求高）
- 哪些场景下，可以有合理的差异？（AI 营养助手开放式对话可以有创意空间，但涉及营养知识的部分须与知识库一致）
- 一致性衡量方式：相同输入重复调用 N 次，比较关键字段的波动范围。设定可接受的波动阈值。

### 4.11 性能与安全规范

**产出物**：性能指标表 + 安全措施清单。

**产出形式**：表格 + 文字。

**受众**：后端开发 / 运维 / 安全 / 法务。

  - 不同场景的响应时间容忍度（信息检查 <3s，单餐分析 <10s，周分析 <20s）
  - 并发与限流：单用户并发数、整体 QPS 上限、超限后的处理策略

  - Prompt 注入防护

## 六、评测体系

### 6.1 评测策略

**产出物**：评测策略总览——离线 vs 在线、自动 vs 人工的适用场景与关系。

**产出形式**：文字（300 字以内）+ 策略矩阵表。

**受众**：AI 工程师 / 测试 / PM 自己。

**写什么**：
- 离线评测：发版前，用固定测试集跑一轮，检查格式合规率和字段填充率。快、便宜、可自动化。
- 在线评测：上线后，用真实用户数据抽样人工评估。慢、贵、但能发现离线评测发现不了的问题（语气、准确性）。
- 两者的关系和配合节奏：每次发版前跑离线，上线后每周抽检在线。

### 6.2 Prompt 评测用例集

**产出形式**：表格。

测试场景覆盖：单餐分析、每日总结、每周分析、AI 助手被动咨询、AI 助手主动触达文案质量。

### 6.3 评测指标

**产出物**：评测指标体系——自动指标表 + 人工评测维度表。

- **v2.0（AI 增强）**：拍照识食、AI 营养助手主动触达（漏餐关怀+建议监督）、RAG 知识库上线，核心指标目标
- **v3.0（生态扩展）**：语音输入、扫码记录、多端同步、订阅制上线、B2B 合作，核心指标目标
