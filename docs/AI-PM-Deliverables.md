## 一、产品概述与 AI 策略

### 产品定位

**食光记**是一个 AI Native 饮食行为优化系统。用户用自然语言记录"吃了什么"，AI 进行单餐即时反馈、每日整体总结、每周长期策略分析。

目标用户：关注健康饮食的普通成年人。MVP 为单用户，数据存浏览器 localStorage。

### 为什么是 AI Native

传统饮食 App 的核心体验是"手动查食物库 → 填克重 → 自动算热量"。这条路有两个致命缺陷：查库太累（用户坚持不了几天），以及只看热量不看结构（一碗面和一碗面加蛋的热量相同，但营养价值完全不同）。

食光记走另一条路：**用户用自然语言描述吃了什么 → AI 估算营养 → AI 给出自然语言反馈**。不追求克级精度，追求可持续使用 + 行为改善。

### AI 的三层角色

对应产品的三个时间尺度：

| 层级 | 时间尺度 | AI 角色 | 输出 |
|------|---------|--------|------|
| 单餐分析 | 即时（每餐） | 营养解读 + 温和反馈 | 营养估算 + ≤20 字评语 |
| 每日总结 | 天 | 整体评估 + 结构分析 | 五项营养状态 + 分析文 |
| 每周分析 | 周 | 趋势发现 + 策略调整 | 周评级 + 下周目标推荐 |

三层之间有递进关系：单餐数据聚合成日，日数据聚合成周。AI 在每一层有不同的分析颗粒度和"说话方式"——单餐只看这一顿，每日不复述每一餐，每周不看每一天。

## 二、AI 能力设计

### 整体链路

```
用户输入饮食描述
       │
       ▼
┌──────────────────┐
│ 阶段一：信息完整性判断 │  ← LLM: 信息够不够分析？
│ 返回: isEnough    │     不够 → 追问（≤20字），等待用户补充
│       assistantMessage │  够   → 进入阶段二
└──────────────────┘
       │ isEnough = true
       ▼
┌──────────────────┐
│ 阶段二：单餐营养分析   │  ← LLM: 估算营养素 + 生成反馈
│ 返回: nutrition   │     前端展示分析卡片
│       estimate    │     用户确认 → 保存 → 回到首页
│       + feedback  │
└──────────────────┘
```

这是一个两阶段 Chain，不是单次调用。拆开的原因：
- **阶段一**是轻量判断，token 少、响应快。如果信息不足，只需返回一句追问，不需要完整的营养分析
- **阶段二**只在信息充足时触发，避免 LLM 基于不完整信息"硬猜"
- 两阶段之间用户可能补充信息，这是多轮对话的自然形式

### 各阶段的 Prompt 设计策略

**阶段一（信息完整性判断）** 的设计要点：

- 明确定义"信息足够"的标准（食物类型 + 大致份量，不要求克重）
- 给出具体例子说明哪些情况需要追问（包子：肉包/菜包？面：汤面/炒面？）
- 约束追问行为：优先补最关键信息，一次最多问两个问题，不重复提问
- `assistantMessage` 硬限制 ≤20 字 → 快速对话，不啰嗦

**阶段二（单餐分析）** 的设计要点：

- 核心约束："只分析这一餐本身"。把"不扩展到全天饮食、不强行挑问题"写进 prompt
- feedback 的调性控制：不是"你这餐太油了！"，而是"下一餐可以配点蔬菜~"
- 输出分为两段：`feedback`（≤20 字，用户一眼看完）和 `analysis_text`（完整分析，想看细节再看）
- 允许估算偏差——"允许基于常见饮食经验进行合理估算，无需绝对精准"

**每日总结** 的设计要点：

- 入参多（用户资料 + 目标范围 + 实际累计值 + 全天菜单文本），但 LLM 的任务明确：只做"今天整体"的分析
- prompt 中显式定义状态判断规则：蛋白质/蔬菜（充足/达标/不足）vs 碳水/油脂/热量（超标/均衡/不足），两套枚举值的区别源于营养学常识——蛋白质和蔬菜"多吃点"通常是好事，碳水和油脂"多吃"通常是问题
- "不逐餐复述"在 prompt 中多次强调（LLM 的天然倾向是做逐条点评，需要强约束）

**每周分析** 的设计要点：

- 最复杂的分析任务：7 天数据 → 趋势 → 下周目标调整
- 输入包含"本周目标范围"+"本周实际平均值"，让 LLM 有对比锚点
- 输出新增 `next_week_target`：这是唯一一个**AI 产出被系统采纳为后续参数**的字段（修改用户的目标范围）
- 调整原则在 prompt 中写得很具体：不足→提高下限、超标→降低上限、达标→维持或微调

### 输出模式选择

早期用 `response_format: { type: "json_object" }` 要求 DeepSeek 输出 JSON。遇到的问题：
- 偶发空白输出
- 字段名偶尔漂移（`calorie_status` 变成 `calories_status`）
- LLM 可能输出 markdown 包裹的 JSON（```json ... ```），前端解析多一道处理

最终改为 **Tool Calling 模式**：定义一个函数让模型调用，函数的 `parameters` 就是期望的 JSON Schema。模型**必须调用**该函数，参数天然符合 Schema。这是利用了 Tool Calling 的副作用——"模型会填好函数参数"——来实现可靠的结构化输出。代价是每次调用多一个 `tool_choice` 参数和 `tools` 数组（约 200-500 token）。

## 三、架构决策记录

### 决策 1：Dify → 自建后端代理

| | Dify 方案（原） | 自建代理方案（现） |
|------|------|------|
| AI 平台 | Dify Chatflow/Workflow | DeepSeek API 直调 |
| 调用方式 | 前端 → Dify API | 前端 → Next.js API Route → DeepSeek |
| API Key | 暴露在前端（硬编码） | 存在服务端环境变量 |
| 可控性 | Dify 黑盒 | 完全掌控 prompt + schema |
| 成本 | 免费额度有限 | 按 DeepSeek token 计费 |

**决策原因**：
1. **API Key 安全**：前端直连 Dify 意味着 Key 暴露在浏览器，任何人都能用。自建后端代理后 Key 永远不出服务器。
2. **Prompt 迭代效率**：Dify 上改 prompt 需要在 Web UI 操作；代码里改 prompt 可以用 GitHub 管理版本、diff 比对、回滚。
3. **成本与控制**：Dify 的免费额度不适合长期使用，DeepSeek 的 API 定价极低，且缓存命中后更便宜。

### 决策 2：Tool Calling 优于 response_format

对 DeepSeek 实测发现 `response_format: { type: "json_object" }` 不够可靠（偶发空白输出、字段名漂移）。Tool Calling 的 `tool_choice: { type: "function" }` 是强制性的——模型**必须**调用指定函数，不调用就是 API 错误。这比 response_format 的"建议"语义更强硬。

### 决策 3：localStorage 而非数据库

MVP 单用户场景下，localStorage 是零成本的正确选择。不需要：
- 数据库部署和维护
- 用户认证系统（localStorage 天然隔离）
- 后端 CRUD 接口（storage.ts 一个文件搞定）

代价是数据只存本地，清浏览器 = 丢数据。这在 MVP 阶段可接受。

## 四、Schema 设计

### 设计原则

**枚举优先**：AI 对自由的文本输出不可控，枚举值限制在有限集合内，前端才能做一致性渲染。例如 `feedback` 是自由文本（≤20 字），但 `protein_status` 必须是 `"充足" | "达标" | "不足"` 之一，因为这些值要映射到 UI 颜色。

**两套状态枚举，逻辑对称但语义不同**：

| 营养类别 | 高于上限 | 在范围内 | 低于下限 |
|---------|---------|--------|--------|
| 蛋白质、蔬菜 | 充足 | 达标 | 不足 |
| 碳水、油脂、热量 | 超标 | 均衡 | 不足 |

为什么用两套词？蛋白质和蔬菜"高于上限"在绝大多数饮食场景中是好事（多吃了不坏事），用正向词汇"充足"；碳水、油脂、热量"高于上限"是需要注意的信号，用"超标"。

**扁平优先**：所有字段一层展开，不用深层嵌套。`next_week_target.protein_g.min` 而不是 `next_week_target.nutrients[0].range.min`。

**Description 即 prompt**：Tool definition 中每个字段的 `description` 直接告诉 LLM 这个字段该怎么填。例如 `"feedback": { "description": "一句简短反馈，不超过20字，不说教，不制造焦虑" }`。这就是 Schema 设计六原则中的"Description 是 prompt"。

### 核心 Schema：单餐分析响应

```typescript
{
  meal_record: string;        // ≤15字，只描述吃了什么
  summary: {
    protein_content: "较多" | "正常" | "较少";
    vegetable_content: "较多" | "正常" | "较少";
    carbohydrate_content: "较多" | "正常" | "较少";
    oil_warning: "是" | "否";
    calorie_warning: "是" | "否";
  };
  nutrition_estimate: {
    protein_g: number;        // 克
    carbohydrate_g: number;
    oil_g: number;
    vegetable_g: number;
    calories_kcal: number;    // 千卡
  };
  feedback: string;           // ≤20字
  analysis_text: string;      // 完整自然语言
}
```

## 五、AI UX 设计

### 多轮对话与异步等待

单餐分析是一个多轮对话流程：用户可能被追问 1-2 次才完成一次分析。UI 设计为聊天界面：
- 用户消息右对齐（绿色气泡）
- AI 追问 / 分析结果左对齐（白色气泡）
- 分析结果卡片化展示（区别于文本追问）

异步等待时的体验设计：
- 发送消息后立即显示加载状态（"...思考中"），避免用户以为卡住了
- 状态栏显示 `isEnough` 阶段和 `analyze` 阶段的不同 loading 文案

### 兜底与容错

| 异常场景 | 处理策略 |
|---------|--------|
| DeepSeek API 返回非 200 | 前端展示通用错误提示 + 重试按钮 |
| Tool Calling 未调用（极端情况） | `aiClient.ts` 降级：尝试从 `content` 字段解析 JSON |
| JSON 解析失败 | 展示原始文本（至少用户能看到 LLM 说了什么） |
| localStorage 读取异常 | 静默回退到空数据 / null |

降级策略的原则：**展示能展示的，不崩溃。**

### 缓存策略

每日总结分析不是每打开一次首页就重新调用 AI。设计了一个**指纹缓存**：

1. 当日所有 MealRecord 的 ID 排序 → 拼接成指纹字符串
2. 指纹未变 → 直接用缓存的总结（不调用 API，零成本）
3. 指纹变更 + 三餐齐全 → 触发新的每日总结分析

这样用户的首页负载是：频繁刷新时只读缓存，只有加了新餐记录后才触发一次 API 调用。

## 六、质量评估与迭代

### AI 输出质量的评估维度

| 维度 | 评估方式 | 目标 |
|------|---------|------|
| feedback 字数 | 前端自动校验 | ≤20 字 |
| meal_record 字数 | 前端自动校验 | ≤15 字 |
| 分析是否越界 | 人工抽查 | 单餐分析不涉及全天，每日总结不逐餐复述 |
| 状态判断准确性 | 对比前端计算值 | AI 状态 ≠ 前端计算状态时标记 |
| 语气是否自然 | 人工体验 | 无明显说教感 / 焦虑制造 |

### 已知改进方向

- **个性化记忆**：目前每餐分析是独立的，AI 不记得用户上一餐吃了什么。可以在单餐分析的上下文里加入"今天已吃"信息，让 feedback 更有上下文感
- **编辑历史餐记录**：当前只支持查看和删除，不支持修改。这会触发每日总结缓存失效（指纹变了），但逻辑上已支持
- **Web 搜索增强**：对不常见食物（地方小吃、海外菜品），可以考虑用搜索补充营养数据
