## 1. 成本模型

### 单用户月成本估算

以 DeepSeek 当前价格（输入 ¥3/百万 token，输出 ¥6/百万 token）计算重度用户：

| 场景 | 单次 token 估算 | 日均调用 | 月调用 | 月成本 |
|------|----------------|---------|--------|--------|
| info_check | ~300 in + ~50 out | 3 次 | 90 次 | ¥0.01 |
| meal_analysis | ~800 in + ~300 out | 3 次 | 90 次 | ¥0.03 |
| daily_summary | ~600 in + ~200 out | 1 次 | 30 次 | ¥0.01 |
| weekly_analysis | ~2000 in + ~500 out | 0.15 次 | 1 次 | ¥0.001 |
| AI 助手对话 | ~500 in + ~200 out | 5 次 | 150 次 | ¥0.03 |
| 主动触达文案 | ~300 in + ~50 out | 1.5 次 | 45 次 | ¥0.01 |
| content_safety | ~200 in + ~50 out | 3 次 | 90 次 | ¥0.01 |

**重度用户月成本 ≈ ¥0.10-0.15**（约合 $0.014-0.02）

### 规模化成本

| MAU | 假设 60% 为重度用户 | 月 LLM 成本 | 年成本 |
|-----|-------------------|-----------|--------|
| 1,000 | 600 | ¥60-90 | ¥720-1,080 |
| 5,000 | 3,000 | ¥300-450 | ¥3,600-5,400 |
| 10,000 | 6,000 | ¥600-900 | ¥7,200-10,800 |
| 50,000 | 30,000 | ¥3,000-4,500 | ¥36,000-54,000 |

**结论**：DeepSeek 定价下，LLM 成本在规模化前不是主要瓶颈。真正的大头是服务器+带宽+人力。到达 5 万 MAU 前无需优化 token 成本。

### 降本触发线

当月度 token 费用超过 ¥5,000 时启动降本：info_check 切轻量模型、daily_summary 缓存命中率优化、AI 助手上下文截断策略收紧。

## 2. Prompt 版本管理与回滚

### 为什么 Prompt 需要版本管理

Prompt 变更的影响面比代码更难预测——代码有类型检查，Prompt 没有。改了 info_check 的追问策略，可能一周后才发现 is_enough 准确率掉了 5%。

### 版本策略

```
Prompt 版本号 = {场景}_{日期}_{变更序号}，如 info_check_20260528_01
```

| 环节 | 规则 |
|------|------|
| 存储 | 所有 prompt 存 Git，与代码同仓库，`prompts/` 目录独立 |
| 变更记录 | 每次变更写清楚：改了什么、为什么改、预期影响 |
| 发版审批 | info_check / meal_analysis / content_safety 变更须 PM + 技术双签；其余 prompt 技术自签 |
| 灰度 | prompt 变更先在 5% 用户上跑 48h，核心指标无显著劣化再全量 |
| 回滚 | 保留最近 3 个版本的 prompt。回滚操作等同于代码回滚，走 Git revert |

### 回滚触发条件

| 指标 | 阈值 | 动作 |
|------|------|------|
| 格式合规率 | 下降 > 3% | 自动告警，PM 决定是否回滚 |
| is_enough 准确率 | 下降 > 5% | 人工抽检确认后回滚 |
| 用户不满意率 | 单日上升 > 10% | 排查原因，疑似 prompt 问题则回滚 |
| 有害内容拦截率 | 下降 > 1% | 立即回滚，无需确认 |

### 面试话术

"Prompt 是 AI 产品的生产配置，应该享受和代码同等的版本纪律。我设计的原则是：高频调用场景的 prompt 变更双签+灰度，安全相关 prompt 的劣化零容忍直接回滚。"

## 3. AI 冷启动体验

### 问题定义

新用户前 3-7 天 memory 为空，AI 个性化最弱，但此时恰是用户决定留不留的关键窗口。不能让用户觉得"这 AI 和普通 App 没什么区别"。

### 分层降级策略

| 记忆层级 | 冷启动时可用？ | 降级方案 |
|---------|-------------|---------|
| 身体数据 + 目标 | 注册后即可用 | 无需降级 |
| 饮食偏好（忌口/素食等） | 首次记录时追问 | info_check 追加一问："有没有不吃的东西？"只问一次 |
| 生活习惯（应酬/加班） | 3-5 天后才有 | AI 标注"刚认识你，先按通用建议来"，明示不确定性 |
| 历史饮食模式 | 7 天后才有 | 周报页标注"再记录几天，我就能更懂你" |

### 关键设计

**首日体验亮点前置**：第一次记录完立刻出营养卡片，让用户在 30 秒内感受到 AI 的价值——不说克重、给行动建议。这个价值不依赖 memory。

**透明度**：当 AI 因为缺乏记忆而给通用建议时，明说"目前是按一般情况建议的，等你多记几天，我能给更准的反馈"——设定合理预期比假装什么都懂更重要。

**渐进式个性化**：

```
第 1 天：基于目标（减脂/增肌/维持）→ 30% 个性化
第 3 天：+ 饮食偏好 → 50% 个性化
第 7 天：+ 生活习惯 + 饮食模式 → 80% 个性化
第 30 天：+ 长期趋势 → 95% 个性化
```

### 面试话术

"冷启动是 AI 产品的体验悬崖。我的策略是：首日价值不依赖 memory（记录→分析→行动建议），让用户先用起来；然后通过追问和自动提取渐进式丰富记忆。同时透明告知 AI 目前的了解程度，管理预期。个性化不是 0 或 1，是随着数据积累逐步解锁的。"

## 4. 用户纠错与反馈闭环

### 当前问题

PRD 里的反馈链路：用户打分（满意/不满意）→ PM 抽样 Bad Case → 归因 → 修 prompt。这个链路最短也是周级别。用户今天发现蛋白质估错了，没法当场纠正。

### 设计补充

**即时纠错入口**（P2，v1.1）：

营养分析卡片右下角增加"纠错"按钮（不显眼但可找到）。点击后弹出快速纠错面板：

| 纠错类型 | 用户操作 | 系统响应 |
|---------|---------|---------|
| 食物识别错误 | "我没有吃红烧排骨，是糖醋排骨" | 触发 meal_analysis 重新分析，覆盖原结果 |
| 份量估算偏差 | "米饭其实是半碗不是一碗" | 允许用户调整份量描述，触发重新估算 |
| 营养素数值明显偏差 | "蛋白质估低了，应该有 40g" | 记录纠错但不覆盖——用户不一定对，标记进入标注队列 |

**纠错数据的使用**：

- 用户主动纠错的案例优先级高于随机抽样，进入 Bad Case 标注队列
- 同类纠错累计 ≥ 5 条 → 自动提升为该场景的优先修复项
- 不将用户纠错数据用于模型训练（合规）

**为什么不做实时覆盖**：用户对营养素的判断可能不准。设计为"记录+标记"而非"直接覆盖"，让专业标注员确认后再决定是否修 prompt。

### 面试话术

"反馈闭环的速度决定 AI 产品的迭代效率。满意/不满意按钮是宏观信号，用来监测趋势；用户主动纠错是微观信号，用来定位具体问题。两者互补——前者回答'有没有问题'，后者回答'哪里有问题'。"

## 5. 竞争护城河

### 这个产品为什么不能轻易被抄

**大模型人人能调，这没错。但以下三点不是：**

**第一，垂直领域的数据飞轮。**
每一次用户保存记录 → AI 分析 → 用户确认或纠正，都是一次标注。100 个用户 × 3 餐 × 30 天 = 9000 条/月带标签的中式餐饮营养估算数据。这些数据可以：
- 微调出中式餐饮专用的营养估算模型（通用模型估西餐准、估中餐飘）
- 构建中式菜品-营养素映射库（"西红柿炒蛋"到底多少油，取决于食堂厨师的心情，但数据多了就有统计规律）

**第二，用户记忆的迁移成本。**
当一个用户在食光记上用了 30 天，AI 记住了 ta 是蛋奶素、应酬多、减脂中、食堂党——这些记忆不是数据导出能带走的。换一个 App 意味着重新"教会"AI 自己是谁。30 天的迁移成本就是护城河的一砖一瓦。

**第三，细分场景的体验纵深。**
大厂可以做"AI 饮食分析"，但不会把 prompt 打磨到"应酬型用户不劝少喝酒、从非应酬日补蔬菜"这种颗粒度。这种场景纵深来自对垂直用户的理解，不是技术壁垒，是认知壁垒。

### 壁垒建设节奏

```
MVP（0-6 月）：体验壁垒 —— 场景纵深 + prompt 打磨 + 用户记忆积累
成长期（6-12 月）：数据壁垒 —— 垂直标注数据集 + 微调模型
成熟期（12 月+）：网络效应 —— 用户量 → 数据量 → 模型更准 → 体验更好 → 更多用户
```

### 面试话术

"AI 产品的护城河不在模型本身，模型是 Commodity。壁垒在三个地方：一是垂直场景的数据飞轮——每一次用户交互都在产生训练数据；二是用户记忆的迁移成本——AI 用越久越懂你，越离不开；三是细分场景的体验纵深——大厂不会为一个'应酬场景'单独优化 prompt，但这恰恰是用户最强的感知差异。前两个是数据壁垒，第三个是认知壁垒。"

## 6. SLA 与运维

### 为什么需要 SLA

PRD 里定义了性能目标（6.4），但目标是期望值。SLA 定义的是"达不到怎么办"——这是 PM 和工程团队之间的接口契约。

### SLA 定义

| 指标 | SLO（服务水平目标） | SLA（达不到的后果） |
|------|-------------------|-------------------|
| AI 接口可用性 | 月度 ≥ 99.5%（月故障时长 < 3.6h） | 低于 99% 时启动根因分析 + 24h 内输出改进方案 |
| info_check P95 延迟 | < 5s | 连续 2h 超阈值 → 技术负责人告警 |
| meal_analysis P95 延迟 | < 10s | 同上 |
| 格式合规率 | ≥ 98% | 单日 < 95% → 冻结 prompt 发版，排查到恢复为止 |
| 有害内容拦截率 | ≥ 99% | 单日 < 98% → 立即告警，人工审核介入 |

### 告警分级

| 级别 | 定义 | 响应 | 通知 |
|------|------|------|------|
| P0 | AI 服务完全不可用，所有用户受影响 | 15min 内响应 | 技术负责人 + PM |
| P1 | 核心场景（记录/分析）延迟超阈值或成功率骤降 | 30min 内响应 | 技术负责人 |
| P2 | 非核心场景异常（周报生成慢、主动触达失败率高） | 2h 内响应 | 开发值班 |

### 兜底与用户沟通

| 故障时长 | 对外沟通 |
|---------|---------|
| < 30min | 不通知，静默恢复 |
| 30min-2h | App 内 toast："AI 分析有点慢，请稍候" |
| > 2h | 首页公告条 + 说明原因 + 预计恢复时间 |
| > 6h | 推送通知 + 事后邮件道歉（如果已收集邮箱） |

### 面试话术

"SLA 不是技术文档，是 PM 对用户的承诺。关键不是定义了多少个 9，而是定义清楚：什么时候告警、谁响应、多长时间、用户看到什么。MVP 阶段不做 99.9%，因为成本不值得——剩下的 0.4% 用用户沟通补上。"

## 附录：面试中最可能被追问的三个问题

### Q1：你怎么确保 AI 不给用户错误的饮食建议？

三层防控（Tool Calling Schema 约束 → 后端数值校验 → 场景级降级），加两条硬红线：不诊断疾病、不推荐极端饮食。加上内容安全两级审核（关键词实时 + LLM 异步语义）。最终防线：每条分析标注"建议仅供参考"。

### Q2：MVP 做了，数据不好怎么办？

先看哪类指标不好——业务指标（留存/记录率）还是 AI 质量指标（准确率/采纳率）。业务不好看场景（是不是自然语言输入也没降低门槛？还是用户记了但觉得没价值？），AI 不好看 Bad Case 分类（格式/准确度/语气/追问），定位到具体场景-问题类型-Prompt/Schema，修完之后跑回归测试集确认不退化再上线。核心逻辑：数据不好的时候不猜，埋点和 Bad Case 机制就是为了告诉你具体哪里不好。

### Q3：这个产品的商业化和用户体验怎么平衡？

核心原则：对用户的核心价值（记录快、反馈有用）永远免费。可收费的点在"更深入"而非"更基础"：进阶营养报告（月度深度分析）、定制化饮食计划（与注册营养师合作的增值服务）、更丰富的知识库访问。广告不碰——饮食健康领域出现广告直接摧毁信任。MVP 阶段不考虑商业化，先把留存做到 25% 以上。

### Q4：你怎么评估一个 prompt 改动的效果好不好？

这本质是 AI 产品的"测试"问题。我的评估体系分三层：

**离线评估（改完立刻跑）**：维护一个 50-100 条的黄金测试集（覆盖正常输入、边界输入、对抗输入），改 prompt 后跑一遍，对比输出结构合规率、关键字段准确率、误拒/误收率。这个测试集不参与 prompt 设计——谁写测试集谁不写 prompt，防止过拟合。

**在线评估（灰度 48h 看）**：5% 流量跑新 prompt，对比核心指标——用户采纳率（看了分析后是否继续记录下一餐）、纠错率、满意度打分。这些是"用户用脚投票"的指标。

**长期评估（上线后 2 周）**：看留存曲线是否出现拐点。prompt 改动对留存的伤害往往是延迟的——今天改坏了，一周后才发现用户不来了。

"三层评估的核心逻辑：离线保证'不出错'，在线保证'体验不降'，长期保证'价值不损'。对应的回滚条件也不同——离线劣化直接不回滚（不进灰度），在线劣化 48h 回滚，长期劣化 2 周内分析归因。"

## 7. AI 产品评估体系（Eval）

### 为什么 Eval 是 AI PM 的第一优先级

传统产品改了功能 → QA 测 → 上线。AI 产品改了 prompt → 输出变了 → 你怎么知道是变好了还是变坏了？没有 Eval，prompt 迭代就是盲飞。

### 三层评估体系架构

```
Layer 1: 离线评估（代码级）  → 改完立刻跑，分钟级反馈
Layer 2: 在线评估（用户级）  → 灰度 5%，48h 观测
Layer 3: 长期评估（业务级）  → 上线后 2 周追踪
```

### Layer 1 详解：离线评估

**黄金测试集构建原则：**

| 原则 | 说明 | 示例 |
|------|------|------|
| 覆盖正常输入 | 80% 的用例来自真实用户输入 | "中午吃了一碗牛肉面加了个蛋" |
| 覆盖边界输入 | 10% 的用例测试边界行为 | 只输入"吃了"两个字 |
| 覆盖对抗输入 | 10% 的用例测试安全边界 | "推荐一个能快速减肥的减肥药" |
| 测试集隔离 | 写 prompt 的人不维护测试集 | 防止"按测试集写 prompt"的过拟合 |

**针对食光记各 agent 的离线指标：**

| Agent | 结构指标 | 质量指标 |
|-------|---------|---------|
| info_check | JSON 解析成功率 > 99% | is_enough 准确率 > 90%（人工标注对照） |
| meal_analysis | 5 字段齐全率 100% | 营养素估算误差 < ±30%（vs 营养师标注） |
| daily_summary | feedback 字数 ≤ 30 | 用户满意度打分 ≥ 3.5/5 |
| content_safety | 拦截率 100%（已知攻击模式） | 误拦率 < 5%（正常饮食讨论不被当成违规） |

### Layer 2 详解：在线评估

**灰度策略**：prompt 变更 → 5% 用户 → 48h → 核心指标无劣化 → 全量

**为什么是 48h？** 因为饮食记录是日频行为。24h 不够覆盖"早餐→午餐→晚餐"的完整循环，至少需要一个完整天 + 第二天早上（确认用户次日是否回来）。

**在线观测指标：**

| 指标 | 检测什么 | 劣化阈值 |
|------|---------|---------|
| 记录完成率 | 开始分析 → 保存成功的比例 | 下降 > 5% |
| 纠错触发率 | 用户点"纠错"的比例 | 上升 > 3% |
| 次日留存 | 今天记录了、明天还来不来 | 下降 > 2% |
| AI 助手对话轮次 | 用户是否需要追问才能满足需求 | 上升 > 1 轮 |

### 面试话术

"AI PM 区别于传统 PM 的核心能力之一就是建立评估体系。代码有单元测试，模型有 benchmark，但 prompt 的质量评估必须自己建。我的方法论是三层评估：离线保证不出错，在线保证体验不降，长期保证价值不损。最关键的是黄金测试集必须和 prompt 写作者隔离——不然就是在考自己出的题。"

## 8. 数据隐私与合规

### 为什么饮食数据比一般用户数据更敏感

饮食数据隐含：健康状况（糖尿病饮食 → 患病推断）、宗教信仰（清真/素食）、经济水平（每餐消费）、作息规律（吃饭时间 → 在不在家）。这些在《个人信息保护法》中至少属于"敏感个人信息"的边缘地带。

### 合规设计

| 维度 | 措施 |
|------|------|
| 数据最小化 | 只存分析所需的字段，不存原始对话全文（meal_record_text 保留但 conversation 不持久化） |
| 目的限制 | 用户数据仅用于营养分析，不出售、不共享、不用于训练基础模型（除非用户明确 opt-in） |
| 存储位置 | MVP 用 localStorage（用户端存储，不上传服务器）。需告知用户：清除浏览器数据会导致记录丢失 |
| 删除权 | 设置页提供"一键删除所有数据"，删除后不可恢复 |
| AI 数据传输 | 调用 DeepSeek API 时仅发送当前餐次描述+营养数据，不发送用户身份信息（姓名、手机号等） |
| 隐私政策 | 注册页展示隐私政策摘要（非全文链接），明确告知：AI 分析基于脱敏数据，不会关联到个人身份 |

### 面试话术

"饮食健康类产品的数据敏感性常被低估。一顿饭的记录能推断出健康状况、宗教信仰、经济水平、作息规律。我的设计原则是：能存本地的存本地、能脱敏的脱敏、能给用户控制的给控制。MVP 阶段用 localStorage 本身就是一种隐私优先的设计——数据在用户手里，不在我们服务器上。但这也意味着用户换设备数据会丢，需要在产品里透明告知。"

## 9. 模型选型与切换策略

### 当前选择 DeepSeek 的理由

| 维度 | 评估 |
|------|------|
| 成本 | ¥3-6/百万 token，是目前性价比最高的选择 |
| 中文能力 | 中式餐饮术语理解优于 GPT-4 同级模型 |
| Tool Calling | 支持 function calling，满足架构需求 |
| 合规 | 国内服务，数据不出境 |

### 选型决策矩阵（切换时使用）

| 权重 | 维度 | 评估方法 |
|------|------|---------|
| 30% | Tool Calling 可靠性 | 跑 100 条测试，统计 tool_calls 格式合规率 |
| 25% | 中文餐饮知识 | 50 道中式菜品营养素估算，对比营养师标注值 |
| 20% | 响应延迟 | P50/P95 延迟，≤ DeepSeek 当前水平 |
| 15% | 成本 | 预估月成本增幅控制在 30% 以内 |
| 10% | 上下文窗口 | ≥ 16K token（满足 weekly_analysis 需求） |

### 切换流程

```
候选模型 → 离线评估（100 条测试集）→ 通过？
  → 5% 灰度 48h → 核心指标无劣化？
  → 30% 灰度 1 周 → 成本/延迟无异常？
  → 全量切换
  → 旧模型保留为降级通道（至少保留 1 个月）
```

### 为什么必须保留降级通道

DeepSeek 挂了 → 用户不能分析 → 一天不记 → 可能就不回来了。降级策略：主模型异常 → 自动切备用模型（如 GPT-4o-mini）+ App 内弱提示"AI 引擎切换中，体验可能略有差异"。对用户来说不应该感知到模型切换。

### 面试话术

"模型选型的核心原则：不绑定任何一家。我的架构里 AI 调用封装在 aiClient.ts 一个文件里，换模型只改这一个文件。选新模型不是比 benchmark 分数，而是跑自己的测试集——你的用户说的'一碗牛肉面'和 benchmark 里的英文问答是完全不同的两件事。"

## 10. AI 产品指标体系

### 为什么 AI 产品需要独立的指标框架

传统产品指标（DAU、留存、转化）衡量的是"用户用不用"。AI 产品还需要衡量"AI 好不好"——这两个维度可能打架：AI 分析质量下降，但 DAU 因为搞了个活动反而上升。没有质量指标的产品团队会发现"数据挺好但用户慢慢不来了"。

### 四维指标框架

```
        业务指标（用户用不用）
        ┌──────────────────┐
        │  DAU / MAU       │
        │  次日/7日/30日留存 │
        │  记录完成率       │
        └────────┬─────────┘
                 │
    ┌────────────┼────────────┐
    │            │            │
    ▼            ▼            ▼
质量指标     成本指标      体验指标
（AI好不好） （花多少钱）  （用完爽不爽）
┌──────────┐ ┌─────────┐ ┌──────────┐
│格式合规率 │ │单次成本  │ │满意度打分 │
│准确率    │ │月总成本  │ │纠错率     │
│拦截率    │ │人均日调用│ │采纳率     │
│误拦率    │ │缓存命中率│ │次日回访率 │
└──────────┘ └─────────┘ └──────────┘
```

### 指标间的制衡关系

| 如果只看... | 会导向什么错误行为 |
|------------|-----------------|
| 只看 DAU | 做活动拉新，不管 AI 质量，劣质体验伤害留存 |
| 只看满意度 | prompt 过度讨好用户（"你吃炸鸡也是对的"），违背营养健康定位 |
| 只看成本 | 切更便宜但更差的模型，体验下降导致留存下滑 |
| 只看准确率 | prompt 越来越保守（不确定时选择不回答），用户觉得"这 AI 没什么用" |

**关键洞察**：AI PM 的日常工作就是在这四个维度之间做权衡。不存在"四个都变好"的改动——你要能说清楚这个改动牺牲了什么、换来了什么。

### 面试话术

"传统 PM 看两个指标：用户来不来、留不留。AI PM 要多看两个：AI 好不好、花多少钱。而且这四个指标是互相制衡的。我的习惯是在每次 prompt 改动前先定义'主优化指标'和'不劣化指标'——比如这次改 info_check 的追问策略，主优化的是'信息收集效率'，不允许劣化的是'用户不耐烦率'。没有这个约束，优化就会变成 trade-off 盲飞。"

## 11. Bad Case 系统性归因

### 为什么需要方法论的归因框架

AI 产品出问题，最常见的两种错误反应：一是"prompt 写得不好，改一下"（不看根因就改），二是"模型不行，换个模型"（不看数据就甩锅）。这两种都是猜。归因框架的作用是：从现象追溯到根因，再决定修什么。

### 四步归因法

```
Step 1: 现象 → Step 2: 分类 → Step 3: 定位 → Step 4: 修复
```

**Step 1 — 收集现象**：用户投诉 / 纠错触发 / 满意度低分 / PM 抽样发现

**Step 2 — 分类（打标签）**：

| 大类 | 子类 | 示例 |
|------|------|------|
| 格式错误 | JSON 解析失败、字段缺失、字段类型错误 | meal_analysis 返回的 fat 是字符串 "高" 而不是 "是"/"否" |
| 准确度偏差 | 份量估错、营养素数值明显偏差、食物误识别 | "一碗牛肉面"估成 200g 蛋白质 |
| 语气/策略问题 | 过于说教、过于讨好、追问过多、忽略用户目标 | 用户说要减脂，AI 推荐了高热量食物 |
| 边界行为错误 | 该拒绝的没拒绝、该追问的没追问、误拒正常输入 | 用户说"我今天不想减肥了"，AI 开始长篇说教 |
| 安全相关 | 漏拦截有害内容、误拦截正常饮食讨论 | 用户讨论"生酮饮食"，AI 判定为违规 |

**Step 3 — 定位根因**：

| 归因方向 | 什么情况下是它的问题 |
|---------|-------------------|
| Prompt 不清晰 | 同类型错误反复出现，且不同输入都触发 |
| Schema 约束不够 | 输出格式漂移，字段值不在枚举范围内 |
| 模型能力不足 | 中文理解偏差/常识错误，改 prompt 也修不好 |
| 用户输入异常 | 极端输入（超长/超短/多语言混杂）触发模型退化 |
| 历史对话污染 | 前一轮对话的错误信息被带到了这一轮 |

**Step 4 — 修复并加入回归测试集**：

| 根因 | 修复方式 | 防止复发 |
|------|---------|---------|
| Prompt 不清晰 | 在 prompt 中增加该场景的明确规则 | 该 Case 加入黄金测试集 |
| Schema 不够 | 收紧 enum 或增加 required 字段 | 在离线评估中加入格式合规检查脚本 |
| 模型能力不足 | 考虑换模型或增加后处理逻辑 | 在测试集中标记"模型能力边界"类用例 |
| 用户输入异常 | 增加输入预处理/截断/拒绝逻辑 | 补齐边界输入测试用例 |

### 面试话术

"AI 产品出 Bug 和传统产品出 Bug 最本质的区别是：传统 Bug 的根因是代码，修一行就修好了；AI Bug 的根因可能在 prompt、在 schema、在模型能力、在用户输入——四个可能根因，而且可能叠加。我做 Bad Case 归因的原则是：先分类再定位，禁止看到问题就改 prompt。因为如果根因是模型能力不足，改 prompt 是治标不治本，下次换一种说法又会出问题。"

## 12. AB 实验在 AI 产品中的特殊性

### 为什么传统的 AB 实验方法论不够用

传统 AB：按钮改个颜色 → 实验组 vs 对照组 → 看点击率 → 显著就上线。

AI AB：prompt 改一句话 → 实验组输出质量变了 → 但用户感知延迟 1-3 天（因为饮食分析的价值是累积的）→ 而且 prompt 改动的效果和用户行为之间存在"翻译层"（用户看到输出 → 理解 → 决定是否采纳 → 行动）。

### AI 产品 AB 的三个特殊挑战

**挑战 1：指标的滞后性**

| 改动类型 | 可观测时间 |
|---------|-----------|
| UI 改动 | 实时 ~ 小时级 |
| 输出结构调整 | 小时级 ~ 天级 |
| 分析质量/策略改动 | 天级 ~ 周级 |

饮食分析的 prompt 改得好不好，用户不会当场说"这个分析真好"，而是体现在 ta 明天还来不来。所以 AI 产品质量实验需要更长的观测窗口。

**挑战 2：实验污染**

同一个用户在同一天用不同 prompt 分析早餐和午餐 → 实际体验是混杂的 → 分组的效果被稀释。解决方案：以用户为随机化单位（不是以请求），确保同一个用户在整个实验周期内看到的是同一个 prompt 版本。

**挑战 3：交互效应**

改了 info_check 的追问策略 → meal_analysis 收到的输入质量变了 → daily_summary 的结论也变了。一个 prompt 的改动会沿着 Agent 链路传导。所以：
- 每次只测一个 Agent 的改动（不要同时改 info_check 和 meal_analysis）
- 如果必须同时改多个，用"全开/全关"对照而不是单独对照

### 什么该做 AB，什么不该做

| 该做 AB | 不该做 AB（直接上） |
|---------|-------------------|
| 追问策略调整（影响用户体验） | 明确的 Bug 修复（如 JSON 格式错误） |
| 语气风格变化（可能影响信任感） | 安全拦截规则加强 |
| 分析详略程度变化 | 后端性能优化（只要不影响用户体验） |
| 新功能入口位置 | 合规性改动 |

### 面试话术

"AI 产品的 AB 实验最容易被低估的是滞后性。你改了一个 prompt，用户第二天才表现出行为变化——如果你只看当天的数据就下结论，什么也看不到。所以我的原则是：质量型改动至少观察 48h，策略型改动至少观察 1 周。同时，随机化单位必须是用户，不是请求，因为同一个用户的不同请求之间存在学习效应。"

### Q5：你的产品如果被大厂抄了怎么办？

大厂有更多资源、更多用户、更好的模型。但饮食健康这个领域，有三个东西大厂抄不走：

**第一，垂直数据的厚度。** 大厂做的是"通用饮食助手"，面对的是"用户今天吃了什么"。我做的是"中式食堂党如何在 15 元预算内吃得均衡"——这些细颗粒度的场景理解来自真实用户的交互数据，不是多雇几个工程师就能缩短的。

**第二，用户记忆的深度。** 一个在食光记用了两个月的用户，AI 已经知道了 ta 是蛋奶素、周三四加班晚归吃外卖、减脂目标但管不住周末聚餐、蛋黄不吃蛋白吃、喜欢麻辣但胃不好。这个记忆深度不是数据导出能带走的。用户去大厂 App 的第一天，AI 跟 ta 说"建议每天摄入 60g 蛋白质"——这和你已经磨合了两个月的 AI 说"今天沙拉别放蛋黄酱哦"是两种完全不同的体验。

**第三，大厂的组织逻辑决定了他们做不深。** 大厂 PM 的 KPI 是 DAU，不是"应酬型用户在第 5 天的蛋白质摄入是否改善"。垂直场景做到让人离不开的程度，需要一个 PM 花 6 个月只打磨一个场景——大厂的 OKR 周期不允许。

"所以我不怕大厂抄。怕的是自己做得不够深——如果用户用了两个月还觉得和第一天的体验差不多，那才是真正的失败。"

### Q6：你觉得 AI PM 和传统 PM 最核心的区别是什么？

**传统 PM 定义的是确定性路径。** 用户点击 → 跳转 → 看到页面 → 完成操作。路径是可预期的，QA 测完就知道上线后是什么样。

**AI PM 定义的是概率性行为边界。** 你不能说"AI 在 X 情况下一定会输出 Y"，你只能说"AI 在 X 情况下不应该输出 Z"。PM 的工作从"设计用户路径"变成了"设计 AI 的行为边界 + 在边界内给 AI 自由度 + 在边界外做兜底"。

具体到日常工作，三个最明显的区别：

| | 传统 PM | AI PM |
|---|--------|-------|
| 写需求 | 写 PRD → 工程师实现 | 写 prompt + schema → 模型输出不可精确控制 |
| 做测试 | QA 按用例测 | 自己建黄金测试集 + 跑离线评估 |
| 修 Bug | 改代码 → 发布 | 归因（prompt？schema？模型？输入？）→ 修 → 建回归测试 |

"本质差异就一句话：传统 PM 交付的是确定性产品，AI PM 交付的是被约束的概率性行为。这个转变对 PM 最大的挑战不是技术理解，而是接受'你不能完全控制产品行为'这件事，然后用评估体系代替控制欲。"
